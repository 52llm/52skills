# Subagent 执行模式（有子代理能力时的进阶打法）

> 单会话循环（SKILL.md 正文）永远是保底。子代理模式的收益是：主上下文不被实现细节淹没、
> 每任务新鲜上下文、可用更便宜的模型跑机械任务。没有子代理时跳过本文件。

## 编排骨架：逐任务 Dev↔QA 环

```
对 tasks.md 的每个任务（默认严格按序；仅 [P] 标记且文件不相交的任务可并行派 implementer，
禁止两个 implementer 碰同一批文件）：
  1. 派 implementer 子代理（带任务简报）
  2. 派 reviewer 子代理对照 spec 验收该任务产出
  3. PASS → 下一个任务；FAIL → 带着 findings 回 implementer 重做
  4. 同一任务重试上限 3 次 → 升级到主会话人工介入
全部任务 PASS → 整分支终审（forge-review）
```

- **连续执行**：任务之间不停下来找用户确认（该确认的在 plan 阶段已经确认完）。
- **逐级门禁**：当前任务不 PASS 不开下一个；全部任务不 PASS 不进集成终审。

## 上下文治理（贵的不是模型，是你的窗口）

- **编排方不碰源文件**：主会话只派活、收结果、记台账；改代码永远发生在子代理里——编排器上下文必须增长得缓慢可预期，这是它能撑完整个变更的前提。

- **File-handoff**：任务简报、diff、评审包一律**落成文件传路径**，不贴进派遣 prompt——贴进去的东西会常驻主上下文，每轮都被重读。
- **抗塌缩**：子代理的长产出先写进 scratch 文件再回一行路径；只回 ≤15 行状态摘要（DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT 四态之一 + 要点）。
- **模型分层**：每次派遣显式指定模型，用"能胜任该角色的最便宜模型"——机械实现用轻模型，评审/裁决用重模型；省不省钱看轮次不看单价。实现与评审尽量**换不同模型**——同模型自审共享同一批盲区，去相关比单价更值钱。

## 任务简报模板（派 implementer 用）

```
任务: Txx <原文照抄，含 → 验证>
需求上下文: <对应 Requirement/Scenario 原文；禁止转述——转述即漂移>
约束: 宪法相关条款 / 命中的 standards / 本任务不许碰的范围
产出要求: 实现 + 让「→ 验证」转绿的证据（命令与输出）+ 勾 tasks.md + 记 progress.md
```

**装配职责**（编排方派活前完成，不外包给子代理现场摸索）：

- **将改文件先读**：本任务要动的每个存量文件，装配前通读，简报附三行——现状行为 / 本任务改哪部分 /
  哪些既有行为必须保住。跳过这步是子代理返工的头号成因。
- **隐式需求也是需求**：让功能在现有系统里端到端跑通所需的行为，任务条目没写也是约束，写进简报。
- **经验回传**：上一任务的意外与教训（翻 progress.md）+ 近几个 commit 体现的既有模式，随简报带下去。

## 评审派遣纪律（防污染）

- reviewer 只读 diff 与 spec，**不读 implementer 的自述报告**——报告里的理由不能降低 finding 的严重度。
- 禁止在派遣词里预判（"这里应该没问题"/"最多算 Minor"）——预判 = 给评审递答案。
- 验收双问分离：①符合 spec 吗（Missing / Extra / Misunderstood）②代码质量过关吗。无法从 diff 判断的标「无法验证」，不许猜。

## 台账即真相

progress.md 是唯一进度真相：每任务完成立刻记 `Txx | done | commit | 证据`。
编排方丢失上下文后：重读台账 + git log 恢复位置，**严禁凭记忆重派已完成任务**。
