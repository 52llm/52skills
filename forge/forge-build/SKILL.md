# forge-build — 证据优先的执行循环

> **Iron Law: 没有新鲜验证证据，不许宣称任何形式的"完成/能用"。**
> 未验证先说"完成"不是效率，是撒谎。验证之后又改了代码 → 旧证据全部作废，重跑。

## 第 0 步：装载与续跑

1. `forge recall <模块/关键词>`：过往同域的坑与规范先入上下文；再读 `.forge/constitution.md`、`changes/<slug>/` 全套产物、命中的 standards。
2. **有 progress.md 台账 → 先重读它再动手**。进度只信台账与 git log，不信记忆——上下文被压缩后凭印象重做已完成任务，是已知最贵的失败。
3. 建工作分支（或 worktree 隔离）。`forge check <slug>` 还没过 → 先回上一阶段清零，不带病开工。

## 任务循环（每个任务都走这五拍）

1. **读**：本任务 + 对应 Requirement/Scenario。只做这一个任务，不预支下一个。
2. **红**：行为改动**先写会失败的验证**（测试/可断言的命令），亲眼看它红——没见过红，你不知道它验的是不是对的东西。纯重构/配置类改动：先跑既有验证记录基线。**没有能变红的验证命令，不准写实现**（构造反馈回路本身就是任务的一部分，回路对了 bug 修一半）。测试只落在**约定好的接缝**上——任务的「→ 验证:」就是约定；想测未约定的接缝，先问用户再写。
3. **绿**：写**最小实现**让验证转绿。外科手术纪律：
   - 只碰本任务必须碰的；不顺手重构、不改隔壁代码/注释/格式；风格向周边看齐，哪怕你有更好的品味。
   - 每一行改动都能追溯到任务与需求；单次使用不抽象，不加没人要的可配置性，不给不会发生的场景写防御。
   - 自己的孤儿自己清（你的改动导致失效的 import/变量删掉）；历史死代码不动，提一嘴让用户决定。
4. **验**：跑本任务验证 + 受影响面（改了回调/中间件/共享状态 → 往外追两层跑相关测试）。
5. **记**：commit（一任务一 commit，checkpoint 语义；逐文件 stage，**禁 `git add -A/.`**）；勾掉 tasks.md 复选框；**立即**在 progress.md 追加一行：`Txx | done | <短hash> | 验证命令+关键输出`。

顺手做：**边走边简化**——每完成一个 Phase 问一次删除测试（删掉这个模块，复杂度是消失还是转移到调用方？）；发现 200 行能写成 50 行就重写这 50 行。

## 计划外发现（执行中一定会遇到，按类处置）

- **顺手修**：本任务直接引起的 bug，与缺失的**正确性要件**（错误处理/输入校验/权限检查）——这些是正确性不是加功能，修完记 progress.md。
- **停下问**：架构级变更（新表/换库/破坏 API）；**包安装失败**——失败的名字可能是幻觉或抢注包（slopsquat），禁止自动换相近名，停给用户核实。
- **挂账不碰**：非本任务引起的存量问题 → 记 `.forge/deferred-work.md`，继续干活。拿不准算哪类 → 按「停下问」。

## 卡住协议

- 同一错误改 2 次没好 → 禁止第三次同样改法：退一步重读代码/日志重新定位，或换路线。
- **3 次修复尝试失败 = 大概率不是假设错了，是架构/定位错了**：停下，回 forge-plan 质疑方案，或切 forge-debug 走系统排障。修 bug 期间任何时刻 → forge-debug。
- 触发 plan.md 里的 tripwire（如改到第 3 个计划外文件）→ 停下回 plan，不硬闯。
- 发现计划本身有问题：小偏差记 progress.md 备注后继续；方向性问题必须停——回 forge-plan 走「中途变卦」三步，改计划比硬闯便宜十倍。

## 收工门（全部任务完成后）

依次通过，缺一不可：

1. 全量验证：完整测试套 + lint + 构建，贴真实输出；失败就修，修完重跑全套。
2. `forge check <slug>`：机械门清零。
3. 通读 `git diff`：每一行都能追溯到任务；发现夹带 → 撤掉。
4. 对照 proposal 成功标准逐条给证据。

**借口拦截表**（任何念头冒出即违规）：

| 念头 | 现实 |
| --- | --- |
| 「应该能用了 / 我有信心」 | 信心不是证据。跑，贴输出 |
| 「就改了一行」 | 简单改动最会在生产里翻车 |
| 「刚才测过了」（之后又改过） | 改过即作废。重跑 |
| 「测试太慢先跳过」 | 跳过验证的"完成"是谎报 |
| 「改一下测试/断言它就绿了」 | 绿灯必须挣来：先独立论证测试本身错了，否则改代码 |
| 「输出瞟一眼像是全过」 | 先搜 fail / error / 非零退出码，判定引用输出原行 |
| 验证前就写「完成/Perfect/搞定」 | 成功语气本身就是红灯 |

## 下一步

交 **forge-review** 评审（评审在合并前，不是之后）。评审清零后收尾四选一，明确问用户：
①合并/提 PR ②保留分支待用户自查 ③继续下一个变更 ④丢弃（**要求用户打出 discard 确认**才删）。
最后走 **forge-compound** 复盘沉淀。多 subagent 并行执行的打法见 `references/subagent-mode.md`。
