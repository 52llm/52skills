# forge-debug — 系统化排障

> **Iron Law: 没有根因调查，不许动手修。** 猜一个改一个是最慢的排障法——
> 每次"试试这样行不行"都在污染现场、烧掉你对系统的信任。

## 第 0 步：把错误读完

完整读错误信息、stack trace、日志——**答案经常就写在里面**。然后 `forge recall <错误关键词>`：这坑以前踩过的话，lesson 里有直达电梯。

## 第 1 步：反馈回路（本阶段的全部秘密）

构造一条**能变红的最小复现命令**：跑一下就能看到失败，改一下就能看到变化。

- **硬门：没有复现命令，禁止进入第 2 步。** 在能让 bug 现形之前靠读代码脑补理论 = 在猜。
- 回路越快越好（秒级 > 分钟级）：failing test > 一条 curl > 最小脚本 > 手工点页面。
- 非确定性 bug：目标不是"每次都复现"，是**提高复现率**（循环跑 N 次、放大并发、固定种子）。
- 回路建对了，bug 已经修掉一半。

## 第 2 步：竞争假设（3-5 个，可证伪）

单一归因是排障里最常见的谬误。写下 3-5 个**互相可区分**的假设，每个附一条**能否证它的观察**：

```
H1 缓存未失效 —— 若为真：清缓存后必然消失 → 验证: <命令>
H2 时序竞争   —— 若为真：单线程跑必然消失 → 验证: <命令>
H3 上游数据脏 —— 若为真：mock 上游必然消失 → 验证: <命令>
```

按「验证成本 × 可能性」排序，**一次只验一个**；被否证的划掉留痕（防止绕回去重猜）。
两个假设都像真的 → 找一条**只有其一能解释**的区分性观察。

## 第 3 步：根因回溯

- **绝不在报错处贴创可贴**。报错位置只是尸体被发现的地方——沿数据流/调用链往回追，找到第一次出错的地方（详细技法见 `references/techniques.md`：插桩、git bisect、污染源二分）。
- 每个"为什么"再问一层，直到答案是流程/设计而不是又一个症状。

## 第 4 步：修复与加固

1. **修根因不修症状**；最小 diff，遵守外科手术纪律（见 forge-build）。
2. **回归测试**：把复现命令固化成测试，亲眼看它红 → 修完看它绿。没有回归测试的修复等着复发。
3. 考虑纵深防御：能不能让这类 bug **结构性不可能**（入口校验/类型收紧/断言），而不是这次侥幸修好。
4. Iron Law 收工：完整验证套跑绿 + 复现命令确认不再复现，贴输出。

## 熔断协议

**3 次修复尝试失败 ≠ 第 4 次换个写法**——大概率是定位错了层或架构有问题：

- 停下。写出：已试什么 / 每次观察到什么 / 现在最可能的原因 / 两个候选方向。
- 质疑：我修的是不是正确的层？问题是不是出在方案本身？→ 回 forge-plan 或与用户对齐。
- 用户说「别猜了/好好想想」= 立刻回第 1 步重建回路，不是再猜一次。

## 下一步

坑值得记（查了超过 30 分钟 / 反直觉 / 会再踩）→ **forge-compound** 沉淀 bug-track lesson
（Problem / Symptoms / What Didn't Work / Solution / Why This Works / Prevention）。然后回到中断的阶段继续。
