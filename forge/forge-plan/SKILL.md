# forge-plan — 从 WHAT 到可执行的 HOW

> 计划要写给「热情高涨但品味差、没有项目上下文、讨厌测试的初级工程师」也能照着执行的程度——
> 模糊处不是留白，是把返工外包给未来。

## 第 0 步：复利检索（开工必做）

- `forge recall <关键词>`：过往 lesson 直接变成本计划的约束（"要避免的失败路 / 要跟的模式"）。
- 通读 `.forge/constitution.md` 与 `standards/index.md` 命中的域标准——计划里的每个决策都要对得起它们。
- 读 `changes/<slug>/proposal.md`（+spec.md）。发现 WHAT 还有窟窿 → 退回 forge-spec，不带着窟窿规划。

## 第 1 步：前提挑战与候选方案

1. **前提挑战**：这是不是该解决的问题？有没有更上游的解法？（一句话回答，写进 plan.md）
2. **10x 检查**：有没有多 1 倍工作量、10 倍价值的版本？（有 → 作为一个候选摆出来）
3. **候选方案 ≥2 个**，其中必须同时有：
   - **方案 A（最小可行）**：能满足全部 P1 的最小实现
   - **方案 B（理想架构）**：不欠债的做法
   - 两者**同权**对比——如果正确答案是重写，就说重写；如果最小可行就够，也别为"架构美"加码。
4. **Constitution Check（门）**：裁决方案逐条对照宪法。违反 → 填论证表（违反原则 / 为什么必要 / 更简单的替代为何被拒）；论证不过 → 改方案，不许"先这样"。

## 第 2 步：写 plan.md

- **裁决与理由**：选哪个方案、为什么；若这是在挑战用户已定的方向 → 用决策简报交用户拍板（见 forge-spec 的简报格式），**永不自动改**。
- **Decisions**：只记三类决策——难回退的 / 缺上下文会觉得奇怪的 / 真实取舍的产物。每条：选了什么、为何不选 Y、被拒替代。
- **Risks & Tripwires**：`[风险] → 缓解`；tripwire = 预设的停下信号（例：改到第 3 个计划外文件、同一测试改 2 次仍红）——触发就回本阶段重看，不硬闯。
- 事前验尸一次：假设两周后这个方案失败了，最可能的死因是什么？把前两名现在就写进 Risks。

## 第 3 步：拆 tasks.md

- **tracer bullet 优先**：第一组任务用一条最薄的垂直切片**纵向打穿所有层**（UI→逻辑→存储→回来），让骨架先跑通，再横向补面。接缝（seam）越少越好，理想是一个。
- **宽改造例外**（改列名/换签名这类单一机械变更炸穿全库，垂直切片切不动）：按 **expand–contract** 排——①展开：新形态与旧并存，谁都不破；②分批迁移调用点，按爆炸半径分批（每包/每目录一批），**批批收尾 CI 绿**（旧形态还在所以绿得了）；③收缩：无人再用时删旧形态，排在全部迁移批之后。批做不到独立绿 → 顺序不变但共享集成分支，绿只在最后的集成验证任务兑现。
- 完成标准从 proposal 成功标准**倒推成可观察真值**（用户能看见什么行为、哪条关键连线必须通），
  不只是任务清单——任务全勾但目标没达成，是 AI 交付的常见结局（评审端有四层实存检查对账）。
- 每个任务 **bite-sized**（一次会话内可完成；整组任务以 ~50% 上下文为目标，给意外留余量），格式：`- [ ] Txx [P?] 描述 → 验证: 命令或可观察检查`。
  - 没有「→ 验证:」的任务不是任务（`forge check` 会 FAIL）。
  - `[P]` 只标真正可并行的（不同文件、无依赖）。
  - `[HITL]` 标只有用户在场才能完成的任务（拍板/授权/人工操作）；自主批跑撞上它必须停下上报——替用户把它答掉 = 任务失败。
  - Txx 编号一经分配不再改——台账、评审、复盘都靠它对话。
- 按优先级分 Phase：地基 → P1（打穿）→ P2/P3 → 收尾（全量验证 + 复盘沉淀两条固定任务）。
- 每个 Phase 末尾一行 `**Checkpoint**: …可独立运行验证`——这是可停下交付的锚点，MVP = 只做到第一个 Checkpoint。
- 禁止占位任务：「适当处理错误」「类似 T03」「后续完善」都是计划失败，写具体。

## 第 4 步：analyze（发车前的一致性体检）

对照 `references/analyze-checklist.md` 做六道只读检查（重复/含混/欠说明/违宪/覆盖缺口/前后矛盾），
按 CRITICAL/HIGH/MEDIUM/LOW 列表输出。**CRITICAL（违宪、核心产物缺失、P1 需求零覆盖）不清零不准发车。**
有子代理 → analyze 派**新鲜上下文**只读产物跑（评产物、不评意图：它没见过你起草时的理由，正合适）。
然后跑 `forge check <slug>` 过机械门，最后把 plan+tasks 给用户拍板——拍板同样附深化菜单
（机制见 forge-spec 质量门），计划侧候选：事前验尸 / 十倍简化 / 抽象阶梯 / 钢人法。

## 中途变卦（从 build/review 折返，或需求在途中变了）

触发：tripwire 命中、方向性错误、用户改需求。禁止一边心里改主意一边继续写码——先走三步：

1. **波及分析**：一句话问题摘要 + 点名哪些产物要改（proposal 的范围/成功标准？spec 哪几条
   Requirement？plan 哪个决策？tasks 哪些作废/新增？）。
2. **三条路线明码标价**：就地调整（改任务继续）／回滚（撤销已做的重来）／砍范围（降级保 MVP，
   砍掉的写进 Out of Scope）。各给一行代价，按决策简报格式交用户拍板——变卦属于 User Challenge，
   **永不自动决定**。
3. **先回写产物再继续**：拍板后先改文档、progress.md 记一行变卦与理由，`forge check` 过了才回
   build；改到哪个阶段的产物，就从哪个阶段重新接力。

## 下一步

**forge-build** 执行。执行中小偏差记进 progress.md 备注继续；方向性错误或需求变化 → 走上面「中途变卦」——改计划比硬闯便宜十倍。
